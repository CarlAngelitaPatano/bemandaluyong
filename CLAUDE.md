This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Be@Mandaluyong** — a Flutter civic/tourism app for Mandaluyong City, Philippines. Features a Heritage Church Trail (visit + verify churches to earn a certificate), local news, city services, attractions, a "Homegrown" local-business directory, weather, and a profile/auth system. Android is the primary target (launcher icon / splash screen config is Android-only; iOS/web are disabled in `pubspec.yaml`), though iOS/web/windows/linux/macos platform folders exist from `flutter create`.

## Commands

```
flutter pub get                     # install dependencies
flutter run                         # run on a connected device/emulator
flutter analyze                     # static analysis (flutter_lints, see analysis_options.yaml)
flutter test                        # run all tests (single file: test/widget_test.dart)
flutter test --plain-name "App starts on the welcome screen"   # run one test by name
flutter build apk                   # release Android APK
flutter build appbundle             # release Android App Bundle (Play Store)

dart run flutter_native_splash:create   # regenerate splash screen after changing assets/icon/*
dart run flutter_launcher_icons         # regenerate launcher icons after changing assets/icon/*
```

There is no lint-fix / format script beyond standard `dart format .`. There is no CI workflow in `.github/` for this app (the only `.github/` content is an unrelated `modernize/java-upgrade` scaffold).

Note: `android_backup/` at the repo root is a stray copy of an older `android/` project (Gradle files, launcher icons, etc.) — it is not part of the build. The live Android project is `android/`.

## Architecture

### Flat, single-package structure
All app code lives directly in `lib/` with **no subfolders** — one file per feature/screen (e.g. `heritage.dart`, `dining.dart`, `news_page.dart`). When adding a new feature, follow this convention rather than introducing a nested folder structure.

### No backend server or database
There is **no Firestore, no custom API, no CMS**. Content is either:
- **Hardcoded Dart data** in the feature file itself (e.g. `kChurches` in `heritage.dart`, city service links in `city_content.dart`, eateries in `dining.dart`).
- **Fetched live from free public APIs with no auth key**: Open-Meteo for weather (`weather.dart`), Google News RSS parsed with the `xml` package (`news_page.dart`), OpenStreetMap raster tiles via `flutter_map` (`trail_map.dart`, `dining.dart`).
- **EmailJS** (`email_service.dart`) sends the Heritage Trail completion certificate email directly from the client using a public key — this is by design (EmailJS is meant to be called from client apps).

Firebase is used **only for Authentication** (`firebase_auth`) — email/password, Google Sign-In, and phone sign-in (`phone_signin.dart`). `firebase_options.dart` is generated by the FlutterFire CLI (`flutterfire configure`) — don't hand-edit it; regenerate it instead. Project id: `be-mandaluyong-4sight` (see `firebase.json`).

### Local persistence via `shared_preferences` only
Every feature that needs to remember state (theme choice, trail progress, read notifications, "remember me", onboarding-seen) uses `shared_preferences` directly, wrapped in a small static service class. The recurring pattern:
```dart
class SomeStore {
  SomeStore._();
  static const String _key = '...';
  static Future<void> load() async { ... }   // read from SharedPreferences into a static field
  static Future<void> _save() async { ... }  // write back
}
```
Examples: `ThemeController` (`theme_controller.dart`, wraps its state in a `ValueNotifier<ThemeMode>` the root `MaterialApp` listens to), `TrailProgress` (`heritage.dart`), `NotificationService` (`notifications.dart`), `Onboarding` (`onboarding.dart`).

### No state-management package
No provider/riverpod/bloc. State is plain `StatefulWidget` + `setState`, with `ValueNotifier`/`ValueListenableBuilder` used for the one piece of state the whole widget tree needs to react to (the active `ThemeMode`, in `ThemeController.instance.mode`). Navigation is plain `Navigator.push(MaterialPageRoute(...))` — no named routes, no `go_router`.

### Demo account special-casing
`kDemoEmail` (`auth_pages.dart`, `demo@bemandaluyong.com`) is a reviewer/demo login that auto-unlocks the entire Heritage Church Trail (`TrailProgress.unlockAll()`). This is checked in three places that must stay in sync when touched: `auth_pages.dart` (on sign-in), `main.dart` `main()` (on cold start with "remember me"), and `HomeShell.initState` (`main.dart`). "Remember me" / auto-login logic also lives in `main()` and gates on `user.emailVerified || isDemo || phoneNumber != null || googleProvider`.

### Design system (`theme.dart`)
All screens should build on the shared tokens rather than hardcoding raw values:
- `AppSpacing` — 4pt spacing scale (`AppSpacing.s/m/l/xl/...`)
- `AppRadius` — corner-radius scale (`AppRadius.sm/md/lg/xl/pill`)
- `AppTheme` — brand colors sourced from the Mandaluyong flag (blue, seed color) and seal (gold accent, city red for identity moments only), Material 3 `ColorScheme.fromSeed`, Google Fonts (Fraunces for brand/display headings, Plus Jakarta Sans for body/UI text), and a semantic `success`/`successFor(brightness)` color for trail-verification moments.

### Motion (`motion.dart`)
Reusable entrance-animation widgets used app-wide to make screens feel alive: `Reveal` (fade + slide-up, staggered with `delayMs`) and `PopIn` (springy scale-in). New list/grid content should cascade in using `Reveal` with incrementing `delayMs` per item, matching the existing pattern in `main.dart`'s `HomePage` and `notifications.dart`.

### Notifications are two distinct systems — don't conflate them
- **In-app notification center** (`notifications.dart`, `NotificationService`): a list *derived on the fly* from current app state (trail progress, weather, static News/Events pointers) — not stored events. Read/unread state is the only thing persisted (`shared_preferences`).
- **OS-level local notification** (`local_notifs.dart`, `LocalNotifs`): a single real daily 7:00 AM Asia/Manila reminder scheduled via `flutter_local_notifications` + `timezone`, set up once in `main()`. Rescheduling replaces (stable id `1001`), never duplicates.
